## 2) 结构性重构议题（原审查项：PracticeOverlay 过大 / 双路 settings / any 类型 / Roam API 封装）

### 现状
- `PracticeOverlay` 仍承担过多职责（渲染、交互、设置、历史清理等）。
- 设置存在双通道（`useSettings` 与 overlay 内 `localSettings`），同步逻辑复杂。
- 数据访问层仍有 `any` 和对全局 `window.roamAlphaAPI` 的直接依赖。

### 风险评估
- 安全风险：中（复杂状态同步更容易引发错误写入与误操作）。
- 性能风险：中（组件过大带来渲染与状态维护负担）。
- 维护风险：高（功能新增时边界不清，回归成本高）。

### 是否建议与 bugfix 同做
- 建议：**不要混做**。
- 理由：该类改动属于架构治理，范围大、回归面广，应单独立项与分阶段发布。

### 推荐立项拆分（给接手工程师）
- 里程碑 A：拆分 `PracticeOverlay`（设置、历史清理、头部状态区分离）。
- 里程碑 B：统一设置源（明确单一 source of truth，清理事件同步冗余）。
- 里程碑 C：引入 `RoamApiClient`（封装读写与错误处理，先从高频接口开始）。
- 里程碑 D：收敛 `any`（以查询层和模型层为优先）。

### 建议验收标准
- 每个里程碑都提供最小回归测试集（优先评分保存、模式切换、历史清理）。
- 所有重构 PR 要求“行为不变更声明 + 风险回滚点”。
- 每个里程碑可独立回滚，不依赖下一阶段上线。
